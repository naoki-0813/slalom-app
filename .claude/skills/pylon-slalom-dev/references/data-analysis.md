# 分析メトリクス・比較アルゴリズム

analysis.js に実装する計算の詳細。単位は内部計算 m/s²・deg/s・m/s、表示は G・deg/s・km/h。

## 1. 前処理

1. **平滑化**: 加速度・角速度に移動平均(窓 約100ms分。サンプル数はレート実測から決定)。
   生データはCSVにそのまま残し、平滑化はメトリクス計算・グラフ表示のみに適用
2. **区間切り出し**: 発進 = speed > 1.4m/s(5km/h) になった最初の時刻、
   停止 = その後 speed < 0.56m/s(2km/h) が3秒続いた開始時刻。
   GPS speed が欠損する端末では、前後加速度の積分速度で代用(誤差が出るが区間検出には十分)
3. **進行率正規化**: 切り出した区間を 0〜100% とし、線形補間で **200点** に
   リサンプリングする。比較・スコア計算はすべてこの200点配列で行う
   - 横軸を時間でなく進行率にする理由: 走行ごとにタイムが違うため、
     時間軸のままでは波形がずれて比較にならない

## 2. 単走メトリクス(結果画面のサマリー)

| 名称 | 計算 |
|---|---|
| 区間タイム [s] | 停止時刻 − 発進時刻 |
| 最大横G [G] | max(\|ay_veh\|) / 9.81(平滑化後) |
| 平均速度 [km/h] | 区間距離(GPS累積 or 速度積分)/ タイム × 3.6 |
| 切り返し回数 | ヨーレート(平滑化後)の符号反転回数。±3deg/s のデッドバンドを設け、ノイズでの誤カウントを防ぐ |
| スムーズネススコア [0-100] | `jerkRms = RMS(d(ay_veh)/dt)` を計算し、`score = 100 * exp(-jerkRms / K)`。K は実走データで調整する定数(初期値 8.0 m/s³)。値が仮であることをコードコメントに明記 |

## 3. お手本(熟練者)比較

基準走行(isExpert=true)と対象走行の200点配列を比較する。

1. **波形差**: `rmseLatG = sqrt(mean((ayMe[i] - ayExpert[i])^2))` を横G・ヨーレートで計算
2. **一致度スコア [0-100]**: `100 * max(0, corr)` (ピアソン相関。横G波形)と
   `100 * exp(-rmseLatG / K2)`(K2初期値 1.0 m/s²)の平均
3. **切り返しタイミング差**: 双方のヨーレートのゼロクロス位置(進行率%)を順に対応付け、
   平均ずれ%を出す。「+の値 = お手本より切り返しが遅い」と表示
4. **タイム差・最大横G差**: 単純差分

講評コメントの生成ルール(if文で十分。AIは使わない):

- タイミング差 > +3%: 「切り返しがお手本より遅れがちです。次のパイロンを早めに見ましょう」
- rmseLatG が大きく最大横Gは同等: 「Gの出し方の波形が違います。舵の当て方が急かもしれません」
- スムーズネス差 < -15点: 「操作が急です。ハンドルをより滑らかに」
- 全スコア良好: 「お手本にかなり近い走りです」

## 4. 再現性スコア(自分の走行同士)

選択された n 本(2〜5本)の200点横G配列について:

1. 各進行率点での標準偏差を計算 → `sdCurve[200]`
2. `meanSd = mean(sdCurve)`
3. `再現性スコア = 100 * exp(-meanSd / K3)`(K3初期値 0.8 m/s²)
4. グラフには平均波形±1σの帯(バンド)を表示し、ばらつきの大きい区間を視覚化する
5. 全ペアのタイムの変動係数(CV)も「タイム再現性」として併記

## 5. グラフ仕様(charts.js)

- ライブラリ: Chart.js(cdnjs、type: 'line')
- 結果画面: 横軸=時間[s]。横G・ヨーレート・速度の3枚(または切替タブ)
- 比較画面: 横軸=進行率[%]。基準走行は太線、他は細線+半透明。
  再現性モードでは平均±1σの帯を `fill` で描く
- 描画データは最大500点程度に間引く(`samples` から等間隔サンプリング)
- グラフのタップ/ドラッグ位置 → 動画シーク連携(結果画面のみ)。
  Chart.js の `onHover`/クリックイベントから x 値を取得する
- 色: 基準=濃い青、自分=オレンジ系。線色は run ごとに固定し凡例表示

## 6. テスト方針

- 上記の計算関数はすべて **純関数**(入力配列→出力値)として書き、DOMに依存させない
- mock.js の「上手い走行」「ばらつき走行」プリセットで:
  - 上手い×2本の再現性スコア > ばらつき×2本の再現性スコア になること
  - 上手い vs ばらつき のお手本比較で rmse が有意に出ること
- 簡易テストページ `test.html`(またはコンソール実行関数)を用意し、
  プリセット波形でメトリクスが期待範囲に入るか一括確認できるようにする
- K, K2, K3 の定数は実走データ収集後にユーザーと見直す前提。マジックナンバーには
  必ず「調整対象」コメントを付ける
